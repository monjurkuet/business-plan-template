# System Architecture — Business Plan Template

> A self-evolving research system for generating, maintaining, and refreshing structured business intelligence across multiple sectors in the Bangladesh market.

---

## 1. System Overview

The Business Plan Template repo started as a collection of manual business-plan templates and evolved into a **self-evolving research pipeline** that:

- Maintains 8 sector directories, each with a consistent set of market-intelligence documents
- Runs an automated 12-step pipeline (audit → freshness → search → extract → score → update → archive → validate → commit → learn)
- Uses **search.datasolved.org** (DuckDuckGo proxy) for web/ news searches
- Routes LLM calls through **llm.datasolved.org** (OpenAI-compatible gateway) with automatic model fallback
- Stores evidence as timestamped JSON artifacts in `data/evidence/`
- Maintains a query memory that learns which search templates yield high-confidence facts
- Archives stale content before refreshing it

**Purpose:** Continuously gather, validate, and synthesise market intelligence so each sector directory stays fresh, accurate, and actionable — without human intervention.

## 3. Data Flow — The Pipeline

The pipeline is orchestrated by `_system/scripts/run_pipeline.py`, which runs 12 scripts in 4 phases:

```
PHASE 1 — AUDIT & VALIDATE
  ├── 1. audit_repo.py
  │     → Scans all sectors/, data/, _templates/, _guides/
  │     → Produces file inventory (json) and audit results (missing files, stale years, duplicates)
  │     → Output: _system/state/file_inventory.json + audit_results.json
  │
  ├── 2. evaluate_freshness.py
  │     → Reads audit_results.json + freshness-policy.yaml
  │     → Classifies each file as fresh/stale/critical based on category thresholds
  │     → Builds a priority queue (P0/P1/P2) of items needing refresh
  │     → Output: _system/state/freshness_queue.json
  │
  └── 3. validate_markdown.py
        → Validates YAML frontmatter (sector, last_verified, freshness, confidence)
        → Checks heading hierarchy, broken internal links
        → Output: _system/state/validation_results.json

PHASE 2 — SEARCH, EXTRACT & SCORE
  ├── 4. execute_searches.py
  │     → Reads freshness queue + search-taxonomy.yaml + sectors.yaml
  │     → Generates search queries (keyword + template substitution)
  │     → Executes via search_client (search.datasolved.org /ddgs)
  │     → Saves raw evidence as data/evidence/{timestamp}.json
  │
  ├── 5. extract_facts.py
  │     → Loads raw evidence from latest data/evidence/ file
  │     → Calls LLM (extract_facts.md prompt) to extract structured facts
  │     → Batches evidence for cost efficiency (batch size = 10)
  │     → Updates evidence file with extracted_facts arrays
  │
  └── 6. score_confidence.py
        → Reads evidence + source-rankings.yaml + scoring-rules.yaml
        → Computes composite confidence score per evidence item
        → Formula: 0.30(source_quality) + 0.20(recency) + 0.20(consistency)
        │          + 0.15(extraction_certainty) + 0.10(entity_precision)
        │          + 0.05(query_historical_success)
        → Tags each item with confidence_level (high/medium/low/unverified)
        → Updates evidence JSON with scores

PHASE 3 — PLAN, UPDATE & ARCHIVE
  ├── 7. build_search_plan.py
  │     → Multi-LLM review using 4 review models:
  │       • deep_analysis (deepseek-v4-pro) — strategic gaps, regulatory implications
  │       • breadth_scan (gemini-2.5-flash-lite) — structure/formatting audit
  │       • trend_gap (gemini-3.1-flash-lite-preview) — emerging trends & cross-sector links
  │       • synthesis (mistral-large-latest) — merges all reviews into action plan
  │     → Output: _system/state/search_plan.json
  │
  ├── 8. update_sector_docs.py
  │     → Filters evidence above confidence threshold (default 0.55)
  │     → Groups evidence by sector
  │     → Calls LLM (synthesis.md) to regenerate sector markdown files
  │     → Updates README.md, landscape reports, competitor profiles
  │
  └── 9. archive_stale.py
        → Reads freshness queue for P0 (critical-stale) items
        → Copies those files to _archive/sectors/{sector}/{timestamp}/
        → Preserves originals for the upcoming refresh

PHASE 4 — SIDECARS & COMMIT & LEARN
  ├── 10. generate_sidecars.py
  │      → Extracts markdown → JSON sidecars for competitors, pricing, sentiment, regulatory
  │      → Saves to data/competitors/ and data/sector-research/
  │      → Fast (<1s), no network/LLM calls needed
  │
  ├── 11. generate_changelog.py
  │      → Reads state from audit, freshness, search_plan, evidence
  │      → Appends human-readable entry to CHANGELOG.md
  │
  ├── 12. git commit + push
  │      → run_pipeline.py stages all changes with "auto: pipeline run {timestamp}"
  │      → Pushes to origin/main
  │
  └── 13. update_query_memory.py
         → Reads latest evidence (with scores)
         → Updates data/query_memory.json with per-query success metrics
         → Applies 0.95 decay factor to historical scores
         → Learning loop: future search planning prioritises high-success queries
```

### How Evidence Flows

```
Search API ──→ raw_evidence.json ──→ LLM extraction ──→ scored_evidence.json
    ↑                                                      │
    │                                                      ▼
freshness_queue.json ←── audit_results.json        update_sector_docs.py
    ↑                                                    │
    │                                                    ▼
evaluate_freshness.py                              Sector README.md
    ↑                                                    competitors/*.md
audit_repo.py                                              ... etc.
    ↑
sectors/ (filesystem scan)
```

## 4. Config Reference

| File | Purpose | Key Contents |
|------|---------|-------------|

| `source-rankings.yaml` | Domain trust weights for confidence scoring | 11 source categories (gov_bd 0.95, international_org 0.92, research_firm 0.88, official_company 0.82, global_news 0.76, bd_news 0.73, linkedin 0.68, facebook 0.64, marketplace 0.49, reddit_forum 0.42, unknown 0.35) |
| `cron.env` | Runtime environment | API keys, base URLs, pipeline limits (MAX_QUERIES_PER_RUN, MAX_LLM_COST_USD_PER_RUN), TZ, run_mode |

## 7. Evidence Pipeline

### Query Memory (`data/query_memory.json`)

- Maps search query text → success metrics (`score`, `hits`, `high_conf_hits`, `total_facts`, `sectors`)
- Scores decay by factor **0.95** each pipeline run (older observations carry less weight)
- `build_search_plan.py` uses query memory to prioritise high-success query templates
- Format per entry:

```json
{
  "queries": {
    "\"auto detailing Dhaka\" policy Bangladesh 2026": {
      "score": 0.3853,
      "hits": 385,
      "high_conf_hits": 6,
      "total_facts": 45,
      "sectors": ["automotive-car-care"],
      "first_seen": "2026-05-06T16:12:50+00:00",
      "last_seen": "2026-06-25T03:10:55+00:00"
    }
  }
}
```

### Confidence Scoring Formula

```
confidence = 0.30 × source_quality
           + 0.20 × recency
           + 0.20 × cross_source_consistency
           + 0.15 × extraction_certainty
           + 0.10 × entity_match_precision
           + 0.05 × query_historical_success
```

**Thresholds:** high ≥ 0.75, medium ≥ 0.50, low ≥ 0.25, unverified < 0.25

- `source_quality`: looked up from `source-rankings.yaml` by domain (gov.bd = 0.95, facebook = 0.64, etc.)
- `recency`: linear decay — `1.0 - min(age_days / stale_after_days, 1.0)`
- `cross_source_consistency`: single_source = 0.50, two_sources_agree = 0.80, three_plus_agree = 1.00, sources_disagree = 0.30
- `extraction_certainty`: from LLM's own confidence in the extracted fact (0.0–1.0)
- `entity_match_precision`: exact_name_match = 1.00, partial = 0.70, context_inferred = 0.40, vague = 0.20
- `query_historical_success`: from `query_memory.json` score

### Freshness Policy

| Category | Stale After (days) | Critical After (days) |
|----------|-------------------|----------------------|
| Default | 90 | 180 |
| Pricing | 14 | 30 |
| Policy | 7 | 21 |
| Facebook Handle | 60 | 90 |
| Competitor Profile | 45 | 90 |
| Sentiment | 21 | 42 |
| Seasonality | 180 | 365 |
| Supply Chain | 60 | 120 |
| Funding | 30 | 60 |
| Demographics | 180 | 365 |
| Technology Adoption | 90 | 180 |

Sector overrides exist for **crypto-bitcoin** (policy: 3/7, pricing: 2/7) and **electronics-gadgets** (pricing: 7/14) due to faster-changing conditions in those sectors.

## 8. Conventions

### File Naming

| Entity | Pattern | Example |
|--------|---------|---------|
| Sector directories | `kebab-case` | `automotive-car-care/` |
| Market subdirectory | `kebab-case-[region]` | `bd-market/` |
| Competitor profiles | `YYYY-MM-DD-competitor-slug.md` | `2026-05-06-shikho.md` |
| Research reports | `YYYY-MM-DD-topic.md` | `2026-05-06-full-landscape-report.md` |
| Evidence files | `YYYYMMDD-HHMMSS.json` | `20260625-030921.json` |

### YAML Frontmatter

Every `.md` file under `sectors/` must have:

```yaml
---
sector: automotive-car-care
last_verified: 2026-06-25
freshness: fresh          # fresh | stale | critical
confidence: 0.749
evidence_ids: ['uuid1', 'uuid2']
---
```

Competitor files additionally require `confidence` in frontmatter.

### Stale Detection

- `audit_repo.py` scans for years **2023** and **2024** in file content (outdated references)
- `evaluate_freshness.py` uses `last_modified` timestamps + freshness-policy thresholds
- Files aged past `stale_after_days` → flagged `stale`
- Files aged past `critical_stale_after_days` → flagged `critical` → queued for archive + refresh
- Category is auto-detected from file path keywords (`pricing` in path → pricing category)

## 9. Adding a New Sector

To add a new sector to the system:

### Step 1 — Define the sector in config

Add an entry to `_system/config/sectors.yaml`:

```yaml
my-new-sector:
  display_name: "My New Sector"
  status: active
  priority_weight: 0.7
  keywords:
    en: ["keyword1 Bangladesh", "keyword2 Dhaka"]
    bn: ["বাংলা কীওয়ার্ড"]
  competitor_target: 10
  subsegments: ["subsegment1", "subsegment2"]
  policy_sensitivity: medium
```

### Step 2 — Create the sector directory

```
mkdir -p sectors/my-new-sector/bd-market/{competitors,research}
```

### Step 3 — Copy templates & initialise docs

```
cp -r _templates/01-idea-brief/idea-brief.md sectors/my-new-sector/bd-market/README.md
cp _templates/03-strategy-canvas/strategy-canvas.md sectors/my-new-sector/bd-market/
cp _templates/05-go-to-market/go-to-market.md sectors/my-new-sector/bd-market/
cp _templates/06-risk-register/risk-register.md sectors/my-new-sector/bd-market/
```

Create placeholders for:
- `pricing-guide.md`
- `regulatory.md`
- `seasonality.md`
- `sentiment-analysis.md`

### Step 4 — Run the pipeline

```bash
python _system/scripts/run_pipeline.py
```

The pipeline will:
1. Detect the new sector in `sectors.yaml` + filesystem
2. Schedule searches for its keywords
3. Execute searches via search.datasolved.org
4. Extract facts via LLM
5. Score and populate the sector docs

### Step 5 — Optionally add freshness overrides

If the sector has fast-changing data (e.g., crypto, electronics), add `freshness_override` to `sectors.yaml` and/or entries in `freshness-policy.yaml`'s `sector_overrides`.

## 10. Model Routing

The system uses four distinct LLM roles, each with a primary model, fallback chain, and specific task:

### Review Models (`build_search_plan.py`)

| Role | Primary Model | Fallbacks | Temperature | Max Tokens | Task |
|------|--------------|-----------|-------------|------------|------|
| `deep_analysis` | `deepseek-ai/deepseek-v4-pro` | deepseek-r1-distill-qwen-32b, gpt-5.4-mini | 0.2 | 12,000 | Strategic gaps, regulatory implications, search priorities |
| `breadth_scan` | `gemini-2.5-flash-lite` | gemini-2.5-flash-lite, gpt-5.4-mini | 0.1 | 6,000 | Structure audit — missing files, heading compliance, stale dates |
| `trend_gap` | `gemini-3.1-flash-lite-preview` | gemini-3-flash-preview, gemini-2.5-flash-lite | 0.3 | 6,000 | Emerging trends, cross-sector links, policy weak signals |
| `synthesis` | `mistral/mistral-large-latest` | gpt-5.4, gpt-5.5, gpt-5.4-mini | 0.1 | 10,000 | Merge all reviews → ranked action plan, content update instructions |

### Extraction Model (`extract_facts.py`)

- Primary: `mistral/mistral-large-latest`
- Fallbacks: `mistral/mistral-large-latest` (retry), `gpt-5.4-mini`, `gemini-2.5-flash-lite`
- Temperature: 0.1

### Update Model (`update_sector_docs.py`)

- Uses the `synthesis` prompt route with `mistral/mistral-large-latest`
- Temperature: 0.1

### API Details

- **Base URL:** `https://llm.datasolved.org/v1`
- **Auth:** `OPENPAI_API_KEY` environment variable (loaded from env or `_system/config/cron.env`)
- **Transport:** urllib with custom 308 redirect handler (required by gateway)
- **Streaming cleanup:** Strips trailing `data: [DONE]` artifacts from non-streaming responses
- **Model aliasing:** Normalises model names across providers (`openai/gpt-5.4` → `gpt-5.4`, `google/gemini-2.5-flash` → `gemini-2.5-flash-lite`, etc.)
- **Retries:** HTTP 429/500/502/503/504 are retried up to 2 times with exponential backoff
- **Preflight:** `/models` endpoint probed once at startup to discover available models

### How Model Selection Works

1. `call_llm()` normalises the model name via alias map
2. Tries the primary model; on HTTP 404, tries the next alias in the candidate list
3. Falls back through the ordered fallback_models list from `model-routing.yaml`
4. `build_search_plan.py` additionally runs a `/models` preflight to skip unavailable models
5. `extract_facts.py` and `score_confidence.py` use hardcoded fallback lists in their scripts

## Appendix: Key Design Decisions

1. **No pip/poetry — stdlib only for core:** Uses `urllib.request` not `requests`, custom YAML parser as fallback. Only `pyyaml` is optional. Keeps the pipeline runnable in constrained environments.
2. **File-system state over database:** All runtime state is flat JSON files in `_system/state/`. No DB needed.
3. **Timestamped evidence as immutable audit trail:** Every pipeline run creates a new evidence JSON — old runs are never overwritten.
4. **LLM-over-search architecture:** The system searches the web, then uses LLM to extract facts from results, rather than relying on LLM's training data alone.
5. **Multi-LLM review with synthesis:** 4 parallel review calls followed by a synthesis call produce better search plans than a single monolithic review.
6. **Confidence scoring before action:** Facts below 0.55 confidence are excluded from sector doc updates, preventing low-quality data from polluting the knowledge base.
7. **Archive-before-refresh:** Critical-stale files are preserved in `_archive/` before being overwritten — no data loss.
